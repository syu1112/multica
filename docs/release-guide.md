# Multica 发版手册

本文面向负责合并 PR、执行数据库迁移和发布服务的同事。除非发布负责人另有明确安排，所有发布都遵循本文流程。

## 1. 发布范围与责任

发布负责人需要确认：

- 目标 PR 已通过评审并合并到目标分支；
- 已确定要发布的 commit SHA 或版本标签；
- 变更是否包含 `server/migrations/` 下的迁移；
- 生产环境的配置、数据库备份和回滚权限均已就绪。

不要以本地工作区代码作为生产发布来源。必须使用远程目标分支上可追溯的 commit SHA 或 tag。

## 2. 发布前检查

1. 确认工作区干净，并获取目标版本：

   ```bash
   git fetch origin
   git checkout <target-branch>
   git pull --ff-only origin <target-branch>
   git rev-parse HEAD
   ```

2. 阅读 PR 描述、迁移文件和上线说明，确认是否存在手工配置、数据回填或灰度要求。
3. 执行与变更匹配的测试；至少保留测试命令与结果。
4. 备份生产数据库，并记录备份文件、时间和版本号。
5. 检查生产环境变量已配置。不得在命令行、日志、PR 或聊天记录中输出密钥值。

## 3. 构建与部署顺序

推荐顺序为：**后端迁移能力 → 数据库迁移 → 后端实例 → Web/桌面端 → 验收**。

1. 构建后端和 Web：

   ```bash
   pnpm install --frozen-lockfile
   pnpm build
   cd server && go build ./cmd/server
   ```

2. 使用项目迁移工具执行所有尚未执行的数据库迁移：

   ```bash
   cd server
   go run ./cmd/migrate up
   ```

   迁移工具会按版本顺序执行未应用的迁移，并跳过已经记录的版本。**不要手工只运行某一个 SQL 文件，也不要跳过中间版本。**

3. 滚动重启后端实例，等待每个实例健康。
4. 部署 Web 静态资源或重启 Web 服务。
5. 发布桌面端时，确认桌面端使用的 API 地址与本次后端环境一致。

## 4. 基础验收

1. 后端健康检查返回 HTTP 200：

   ```bash
   curl -fsS https://<multica-host>/health
   ```

2. Web 首页、登录、工作区列表和一个 Issue 详情页可正常打开。
3. 检查后端日志：无迁移失败、数据库连接失败、重复端口监听或持续错误重试。
4. 验证一个常规写操作（例如创建测试 Issue）能够落库，并在页面正常显示。
5. 记录实际发布的 commit SHA、开始/结束时间、执行人和验证结果。

## 5. 本次飞书成员个人 Bot 的额外步骤

本分支包含飞书收件箱通知与成员个人 Bot 能力。发布前后还必须完成以下检查。

### 5.1 必需环境变量

| 变量 | 用途 | 要求 |
| --- | --- | --- |
| `MULTICA_LARK_SECRET_KEY` | 加密保存飞书 Bot 的 `app_secret` | 所有后端实例必须相同、稳定且保密；不能随意更换。 |
| `MULTICA_PUBLIC_URL` | 生成飞书内跳转和绑定链接 | 填写用户可访问的 HTTPS 域名，不能是 `localhost`。 |

### 5.2 数据库迁移

本功能涉及迁移 125–128：

- 125：飞书收件箱通知记录；
- 126：运行时关联的飞书安装记录；
- 127：通知事件类型；
- 128：成员个人 Bot 与工作区通知策略。

新环境会执行所有历史迁移；已有环境只执行数据库中尚未记录的版本。通过 `go run ./cmd/migrate up` 统一执行即可。

### 5.3 功能验收

1. 工作区管理员在“设置 → 飞书”配置通知事件类型并保存。
2. 成员 A、成员 B 分别在“设置 → 飞书”扫码创建自己的 Bot。
3. A 通过 Bot 创建 Issue，创建者必须是 A；B 创建的 Issue 必须归属 B。
4. 给 A 产生已启用的收件箱事件，消息只能由 A 的 Bot 收到。
5. A 的设置页不能看到或操作 B 的个人 Bot。

更详细的飞书专项回滚与排障见：[成员个人飞书 Bot 上线手册](superpowers/runbooks/2026-07-14-member-personal-lark-bot-go-live.md)。

## 6. 回滚

1. 立即停止继续扩容或发布，记录错误日志和当前 commit SHA。
2. 优先回滚应用版本到上一稳定版本；确认健康检查恢复。
3. 只有在确认应用回滚无法兼容新表结构时，才执行对应迁移的 down 操作。数据库回滚前必须再次备份。
4. 对于迁移中已产生的业务数据，先评估数据丢失影响；不要直接执行 down migration。
5. 回滚完成后重新执行基础验收，并将结果同步给发布负责人。
